# Путь изучения интеллектуальной рабочей среды (IWE)

> **IWE (Intellectual Work Environment)** — интеллектуальная рабочая среда, аналог IDE для развития мышления. Как IDE даёт программисту редактор, компилятор, линтер и дебаггер — так IWE даёт человеку формализованные знания (Pack), автоматическое извлечение (Экстрактор), проверку корректности (FPF/SPF) и диагностику пробелов (Digital Twin). Человек работает вместе с ИИ-агентами, каждый из которых играет свою роль.
>
> Каждый раздел: **зачем** → **что изучить** → **где найти**.

## 1. Что такое IWE

### 1.1. Определение

IWE — персональная система для интеллектуальной работы и развития. Как IDE объединяет редактор, компилятор и дебаггер в одну среду для программиста — так IWE объединяет знания, планирование и ИИ-агентов в одну среду для мышления.

### 1.2. Анатомия IWE: четыре архитектурных вида

IWE состоит из элементов четырёх разных типов: системы, описания, роли и артефакты. Каждый тип рассматривается через свой **архитектурный вид** (viewpoint) — стандартный подход к описанию сложных систем (ISO/IEC/IEEE 42010, TOGAF).

> **Три классификации IWE:** Виды (этот раздел) отвечают на «из чего состоит». Контуры L1–L4 (§ 2.1) — «где живёт». Тиры T1–T5 (§ 9.1) — «какой уровень доступа».

#### Вид 1: Системы (U.System) — что работает

Вычислительные системы с границами, входами и выходами. Можно запустить, остановить, обновить. Здесь перечислены основные системы IWE; дополнительные (WakaTime, Digital Twin и др.) описаны в § 2.6.

| Система | Тип | Что делает | Контур (§ 2.1) |
|---------|-----|-----------|--------|
| **Claude Code CLI** | LLM-инструмент | Основной ИИ-исполнитель: код, анализ, планирование | L4 Personal |
| **Telegram-бот** (@aist_me_bot) | Сервис | Заметки, программы, Digital Twin, уведомления | L2 Platform |
| **MCP-серверы** | Протокол | Доступ к Pack, гайдам, DS-описаниям из Claude Code | L2 Platform |
| **Git + GitHub** | VCS | Версионирование, хранение, CI | L3 Template / L4 |

> **Тест:** Система → можно спросить «работает / не работает?», перезапустить, обновить версию.

Роли (Вид 3) запускаются автоматически через системный планировщик ОС: launchd (macOS) или cron (Linux). Планировщик не входит в IWE — это инфраструктура операционной системы. Устанавливается один раз при setup.

#### Вид 2: Описания (U.Description) — чем руководствуется среда

Текстовые описания, которые загружаются в контекст ИИ и определяют его поведение. Не запускаются — читаются.

| Описание | Состав | Назначение |
|----------|--------|-----------|
| **Экзокортекс** | `CLAUDE.md` + `MEMORY.md` + `memory/*.md` | Правила, протоколы (ОРЗ, WP Gate, Capture-to-Pack), различения, SOTA, навигация |
| **Методология** (FPF, SPF) | Закодирована в экзокортексе и промптах | Принципы корректного мышления, fallback chain |
| **Pack-сущности** | `PACK-{область}/pack/**/*.md` | Формализованные описания предметных областей (source-of-truth) |
| **Промпты агентов** | `roles/*/prompts/*.md` | Сценарии: day-plan, week-review, session-close и др. |

> **Тест:** Описание → можно спросить «актуально / устарело?», прочитать, обновить текст.

Методология (FPF, SPF) — это содержание экзокортекса. Протоколы, различения и правила закодированы в файлах `memory/protocol-*.md`, `memory/hard-distinctions.md`, `CLAUDE.md`.

#### Вид 3: Роли (U.RoleAssignment) — кто что делает

Роль описывает функцию (ЧТО делать), а не исполнителя (КТО). Один исполнитель может играть несколько ролей. Одну роль могут играть разные исполнители (Claude, bash-скрипт, человек). Подробнее: [DP.AGENT.001 §3](https://github.com/TserenTserenov/PACK-digital-platform/blob/main/pack/digital-platform/02-domain-entities/DP.AGENT.001-agent-architecture.md).

| Роль | Код | Исполнитель | Что делает | Когда |
|------|-----|-------------|-----------|-------|
| **Стратег** | R1 | Claude CLI (по расписанию) | Планирование, рефлексия, подготовка к сессиям | Каждое утро, вечер, неделя |
| **Экстрактор** | R2 | Claude CLI | Извлечение описаний в Pack | При Close, по запросу, каждые 3 ч |
| **Синхронизатор** | R8 | bash-скрипт (по расписанию) | Координация расписания, уведомления, ночной обзор | По расписанию |
| **Пользователь** | — | Человек | Принятие решений, создание, рефлексия | Всегда |

> **Тест:** Роль → можно спросить «какие обязательства? какие рабочие продукты?»
>
> **Нотация:** `Исполнитель#Роль:Контекст@Окно` (FPF A.2). Полный каталог: 21 платформенная роль в DP.AGENT.001 §3.2.

#### Вид 4: Артефакты (U.Work) — что производится

Наблюдаемые рабочие продукты. Можно прочитать, проверить, версионировать.

| Артефакт | Где | Кто производит | Для чего |
|----------|-----|---------------|----------|
| **Стратегический хаб** | `DS-strategy/` | Стратег + пользователь | Хранение личных документов (планы, стратегия, inbox) и проведение сессий стратегирования |
| **Pack-репозитории** | `PACK-{область}/` | Экстрактор + пользователь | Накопление формализованных описаний предметной области (единственный source-of-truth) |
| **Проектные репо** | `DS-{проекты}/` | Пользователь + Claude Code | Создание конкретных продуктов: код, боты, курсы, контент |
| **Цифровой двойник** | Бот → Neon DB | Бот + пользователь | Персонализация обучения и рефлексия: профиль, прогресс, самооценка, контекст |
| **Заметки** | `DS-strategy/inbox/` | Бот (из Telegram) | Быстрая фиксация мыслей и наблюдений для последующей обработки Стратегом |

> **Тест:** Артефакт → можно спросить «готов / не готов?», показать заказчику.
>
> **Цифровой двойник** — не система, а информационная модель (U.InformationModel). Хранится в системе (бот + Neon DB), но сам является артефактом — набором данных о пользователе.

#### Как виды связаны

```
Роли (Вид 3)           используют →     Системы (Вид 1)
  Стратег ─────────────────────────────── Claude CLI
  Экстрактор ──────────────────────────── Claude CLI
  Синхронизатор ────────────────────────── bash
  Пользователь ─────────────────────────── Claude CLI + бот + VS Code

Роли (Вид 3)           загружают →       Описания (Вид 2)
  Стратег ─────────────────────────────── Промпты + MEMORY.md
  Экстрактор ──────────────────────────── Pack-сущности + протокол
  Пользователь ─────────────────────────── Экзокортекс + методология

Роли (Вид 3)           производят →      Артефакты (Вид 4)
  Стратег ─────────────────────────────── WeekPlan, DayPlan
  Экстрактор ──────────────────────────── Pack-сущности
  Пользователь ─────────────────────────── Код, контент, Pack
```

> **Принцип целостности:** Убери любой вид — и IWE деградирует. Без систем — нет исполнения. Без описаний — stateless ассистент. Без ролей — хаос задач. Без артефактов — нет результата.

### 1.3. Путь пользователя

```
Бот (T1-T3)                     Экзокортекс (T4)                   Созидание (T4+)
├── /start в боте                ├── setup.sh (10 мин)              ├── Свой Pack
├── Марафон, программы           ├── Claude Code + Стратег           ├── DS-проекты
├── Digital Twin                 ├── DS-strategy (планы)             ├── Образовательный контент
└── Персонализация               └── Capture-to-Pack                 └── Масштабирование
```

**Ключевое:** Бот (T1-T3) работает без Git. Экзокортекс (T4) добавляет Claude Code, Git, автоматических агентов. Переход постепенный — всё ранее накопленное (Digital Twin, профиль) сохраняется.

**Центральный инвариант IWE:** Обновления платформы (Standard) **никогда** не затрагивают данные пользователя (Personal). Твои планы, знания и стратегия принадлежат тебе.

## 2. Архитектура: контуры и пространства

### 2.1. Четыре контура системы

IWE существует не изолированно — она часть 4-контурной системы. Каждый контур соответствует своему уровню в иерархии принципов (§ 3.1):

```
L1: Ecosystem    — вся система: платформа + сообщество + все IWE пользователей
  L2: Platform   — инфраструктура и сервисы (бот, MCP, Knowledge Index)
    L3: Template — этот шаблон (CLAUDE.md + memory/ + Стратег + seed/)
      L4: Personal IWE — твой экземпляр (настроенный, с личными Pack и данными)
```

| Контур | Что для тебя | Пример | Как обновляется |
|--------|-------------|--------|-----------------|
| **L1: Ecosystem** | Сообщество, семинары, контент | systemsworld.club, Telegram-каналы | Ты участвуешь |
| **L2: Platform** | Сервисы, к которым подключаешься | Бот @aist_me_bot, Knowledge Index | Обновляется разработчиком |
| **L3: Template** | Шаблон, из которого создан твой IWE | Этот репо (FMT-exocortex-template) | `update.sh` — Platform-space |
| **L4: Personal IWE** | Твоя работа, планы, знания | ~/Github/CLAUDE.md, DS-strategy/ | Только ты (User-space) |

**Где изучить:**
- [ONTOLOGY.md](../ONTOLOGY.md) § «Контуры системы»
- [DS-ecosystem-development/11-platform-contours.md](https://github.com/TserenTserenov/DS-ecosystem-development) — полная архитектурная модель

### 2.2. От шаблона к рабочему пространству

#### Структура репо FMT-exocortex-template

```
FMT-exocortex-template/
│
├── CLAUDE.md                        # Правила для Claude Code
├── README.md                        # Быстрый старт
├── REPO-TYPE.md                     # Тип репозитория (Format)
├── ONTOLOGY.md                      # Онтология экзокортекса
├── update.sh                        # Обновление из upstream
│
├── memory/                          # Оперативная память (≤10 файлов)
│   ├── MEMORY.md                    # ★ PERSONAL: задачи, навигация
│   └── *.md                         # PLATFORM: протоколы, SOTA, чеклисты
│
├── docs/                            # Справочная документация
│   └── LEARNING-PATH.md             # Этот файл
│
├── roles/                          # Роли (точка расширения)
│   └── strategist/                  # Стратег: промпты + скрипты + launchd
│
├── seed/                            # Заготовки → отдельные репо после setup
│   └── strategy/                    # → DS-strategy/
│
└── .claude/                         # Конфигурация Claude Code
    ├── hooks/                       # WakaTime heartbeat
    └── skills/                      # /setup-wakatime
```

#### Три зоны

| Зона | Что | update.sh | Пользователь |
|------|-----|-----------|-------------|
| **PLATFORM** | `memory/*.md` (кроме MEMORY.md), `roles/`, `docs/`, `.claude/` | Обновляет | Не трогает |
| **PERSONAL** | `memory/MEMORY.md` | Не трогает | Редактирует каждую сессию |
| **SEED** | `seed/strategy/` | N/A | После setup → отдельный репо DS-strategy/ |

#### Что делает setup.sh

1. Форкает шаблон → твой GitHub аккаунт
2. Подставляет 7 плейсхолдеров (`{{GITHUB_USER}}`, `{{WORKSPACE_DIR}}` и др.)
3. Копирует `CLAUDE.md` → корень рабочей директории
4. Копирует `memory/*.md` → `~/.claude/projects/.../memory/`
5. Создаёт `DS-strategy/` из `seed/strategy/` (отдельный приватный репо)
6. Устанавливает launchd-агентов для Стратега

#### Workspace после setup

```
~/Github/
├── CLAUDE.md                          # читается каждую сессию (авто)
├── DS-strategy/                       # ★ ежедневно: планы, inbox, стратегия
│   ├── current/DayPlan, WeekPlan      # Стратег пишет, ты читаешь
│   ├── inbox/WP-*.md                  # контексты задач
│   └── docs/Strategy.md              # твоя стратегия
├── FMT-exocortex-template/            # НЕ трогать (обновляется через update.sh)
├── PACK-{область}/                    # когда создашь: доменные знания
└── DS-{проекты}/                      # когда создашь: код, инструменты
```

## 3. Фундамент мышления

### 3.1. Иерархия принципов

Всё знание организовано в 4 уровня. Каждый следующий ограничен предыдущим:

```
Уровень 0: ZP (нулевые принципы)         ← аксиомы, фреймворка нет
    ↓ дисциплинируют
Уровень 1: FPF (первые принципы)          ← принципы + фреймворк (бандл)
    ↓ ограничивают
Уровень 2: SPF → Pack (вторые принципы)   ← фреймворк + принципы (раздельно)
    ↓ определяют
Уровень 3: S2R и др. → DS                  ← фреймворки + принципы (раздельно)
```

**Fallback chain:** DS (3-й) → Pack (2-й) → Base.Принципы (SPF → FPF → ZP). Если непонятно на текущем уровне — поднимись на уровень выше.

**Нулевые принципы (ZP)** — 6 транс-дисциплинарных ограничений:

| Принцип | Суть |
|---------|------|
| ZP.1 Аксиоматичность | Строй на аксиомах, не на интуиции |
| ZP.2 Структура и симметрия | Описывай через инварианты, не объекты |
| ZP.3 Многомасштабность | Модель должна работать на разных масштабах |
| ZP.4 Оптимизация | Ищи экстремум, не перебирай |
| ZP.5 Вероятность и информация | Описывай неопределённость количественно |
| ZP.6 Вычислительные пределы | Учитывай конечность ресурсов |

**Где изучить:**
- [ZP/hierarchy.md](https://github.com/TserenTserenov/ZP/blob/main/hierarchy.md) — карта всех 4 уровней
- [ZP/principles/](https://github.com/TserenTserenov/ZP/tree/main/principles) — каждый принцип подробно
- [CLAUDE.md](../CLAUDE.md) § 1 — таблица типов и fallback chain

### 3.2. Жёсткие различения

22 пары понятий, которые **нельзя путать**. Путаница — главный источник ошибок:

| # | Пара | Суть |
|---|------|------|
| 1 | Система ≠ Эпистема | Физические границы vs. область знаний |
| 2 | Метод ≠ Инструмент | Способ работы vs. инструмент работы |
| 3 | Рабочий продукт ≠ Описание | Наблюдаемый артефакт vs. текст о нём |
| 4 | Учёт ≠ Планирование | Фиксация фактов vs. намерения |
| 5 | Роль ≠ Агент ≠ Инструмент | Маска vs. кто носит маску vs. инструмент |
| 6 | Метод ≠ Навык | Воспроизводимый процесс vs. личная способность |
| 7 | Наблюдение ≠ Суждение | Факт vs. интерпретация |
| 8-10 | Данные ≠ Инсайт, Артефакт ≠ Процесс, Pack ≠ Governance | Онтологические |
| 11 | Процесс ≠ Сервис ≠ Сценарий | Логика vs. точка входа vs. межсистемный путь |
| 12-21 | Описание ≠ Знание, DDD стратегический ≠ тактический, ... | Методологические и операционные |
| 22 | Platform ≠ Template ≠ Personal IWE | 4 контура системы (§ 2.1) |

**Где изучить:**

### 3.3. Первые принципы FPF

FPF (First Principles Framework) — «операционная система для мышления». Определяет базовые конструкции и правила их комбинирования.

| Часть | Содержание | Когда читать |

## 4. Репозитории и проекты

### 4.1. Три типа репозиториев

Каждый репозиторий принадлежит одному из 3 типов. Тип определяет, кто создаёт и что хранит:

| Тип | Подтип | Что хранит | Source-of-truth? | Примеры |
|-----|--------|-----------|-----------------|---------|
| **Base** | Принципы | ZP, FPF, SPF — принципы и фреймворки | Да | ZP, FPF, SPF |
| **Base** | Форматы | FMT-* — протоколы структуры | Да (для формата) | FMT-exocortex-template, FMT-s2r |
| **Pack** | — | Паспорт предметной области | Да | PACK-{область} |
| **DS** | instrument / governance / surface | Производные от Pack | Нет | DS-strategy, DS-ai-systems |

**Ключевое:** **Base = платформа выдаёт** (принципы, фреймворки, шаблоны). **Pack и DS = пользователь создаёт.** Pack — **единственный** source-of-truth для доменного знания. DS потребляет, не создаёт.

**Где изучить:**
- [CLAUDE.md](../CLAUDE.md) § 1 — полная таблица, fallback chain
- [memory/repo-type-rules.md](../memory/repo-type-rules.md) — правила по каждому типу

### 4.2. DS: три подтипа

DS — самый частый тип репозитория, который ты будешь создавать:

| Подтип | Что хранит | Примеры | Когда создавать |
|--------|-----------|---------|----------------|
| **governance** | Планы, стратегия, координация | DS-strategy, DS-ecosystem-development | При setup (DS-strategy — автоматически) |
| **instrument** | Код, боты, агенты, MCP | DS-ai-systems, DS-aist-bot | Когда строишь систему на основе Pack |
| **surface** | Курсы, гайды, посты, контент | DS-Knowledge-Index, DS-blog | Когда создаёшь образовательный контент |

### 4.3. Base/Форматы — стандартные шаблоны

Платформа предоставляет стандартные форматы (Base/Форматы) — протоколы структуры репозиториев:

| Формат | Назначение | Для кого |
|--------|-----------|----------|
| **FMT-exocortex-template** | Персональное рабочее пространство (IWE) | Каждый пользователь T4+ |
| **FMT-s2r** | Проектные репо: 3×3 матрица (системы × роли) | Продвинутые пользователи с многокомпонентными проектами |

**FMT-s2r (System-to-Role)** организует проект по ядрам (kernels), каждое из которых описывается через 9 документов (3 системы × 3 роли). Полезен, когда проект имеет несколько систем: мобильное приложение + бэкенд + инфраструктура.

> **Свои форматы:** Пользователь может создать собственный формат — это будет репо DS с пометкой `template: true` в REPO-TYPE.md.

**Где изучить:**
- [FMT-s2r/README.md](https://github.com/TserenTserenov/FMT-s2r) — обзор и структура

### 4.4. Создание и управление DS-проектами

**Когда создавать:**

| Ситуация | Что создать | Как |
|----------|------------|-----|
| Определил область знаний | `PACK-{область}` | Из шаблона [SPF/pack-template](https://github.com/TserenTserenov/SPF) |
| Строишь систему (бот, инструмент) | `DS-{проект}` (instrument) | `gh repo create DS-my-tool --private` |
| Создаёшь курс или контент | `DS-{проект}` (surface) | `gh repo create DS-my-course --private` |
| Координируешь несколько систем | `DS-{хаб}` (governance) | `gh repo create DS-my-hub --private` |

**Что должно быть внутри каждого DS-*:**
- `CLAUDE.md` — правила для Claude Code (специфичные для этого репо)
- `WORKPLAN.md` — текущие задачи
- `MAPSTRATEGIC.md` — куда двигается ЭТА система

**MAPSTRATEGIC.md vs Strategy.md:**

| | MAPSTRATEGIC.md | Strategy.md |
|---|----------------|-------------|

**Поток:** MAPSTRATEGIC (каждый репо) → Стратег (session-prep) → Strategy.md → WeekPlan

## 5. Повседневная работа

### 5.1. Три стадии сессии: Open → Work → Close

Каждая рабочая сессия с Claude Code проходит три обязательные стадии:

```
Open (вход)           →  Work (работа)            →  Close (выход)
├─ WP Gate            ├─ Capture-to-Pack            ├─ Knowledge Extraction
└─ Ритуал согласования └─ Рубежные проверки         ├─ Обновление MEMORY.md
                                                    ├─ Git commit + push
                                                    └─ Отчёт Close
```

**Пропуск Open** = незапланированная работа. **Пропуск Close** = незафиксированный результат.

#### WP Gate (блокирующее)

**Первое действие на ЛЮБОЕ задание:** проверить, есть ли задача в плане.

1. Прочитать MEMORY.md → секция «РП текущей недели»
2. Совпадает → работаем
3. Не совпадает → СТОП → записать РП → только потом начать

**Исключения:** задачи ≤15 мин, вопросы без изменений, экстренные баг-фиксы.

#### Ритуал согласования

Перед работой Claude объявляет:

> *«Роль: [из таблицы]. Работа: [что]. Рабочий продукт: [артефакт]. Метод: [как]. Оценка: [~Xh].»*

#### Close Protocol (чеклист)

- [ ] Knowledge Extraction (собрать captures)
- [ ] Обновить MEMORY.md (статусы РП)
- [ ] Git commit + push
- [ ] Обновить DS-strategy/current/Plan (статусы)
- [ ] Backup: memory/ → DS-strategy/exocortex/
- [ ] WP Context File: обновить или архивировать
- [ ] Отчёт Close: что сделано, что осталось

#### Exit Protocol (для всех агентов)

| # | Шаг | Зачем |
|---|-----|-------|
| 1 | Артефакт | Без артефакта — работа не существует |
| 2 | Статус | Без статуса — прогресс невидим |
| 3 | Уведомление | Без уведомления — цепочка обрывается |

**Где изучить:**
- [CLAUDE.md](../CLAUDE.md) § 2 — полные протоколы
- [memory/protocol-open.md](../memory/protocol-open.md), [protocol-work.md](../memory/protocol-work.md), [protocol-close.md](../memory/protocol-close.md)

### 5.3. Capture-to-Pack: фиксация знаний

На каждом рубеже (подзадача, паттерн, решение) проверяй — есть ли знание для записи:

| Тип знания | Куда | Когда | Кто пишет |
|------------|------|-------|-----------|
| Правило для всех репо (1-3 строки) | `~/Github/CLAUDE.md` | Сразу | Claude |
| Правило для одного репо | `<repo>/CLAUDE.md` | Сразу | Claude |
| Доменное (архитектура, паттерны) | Соответствующий Pack | При Close | Экстрактор |
| Различение, метод, FM | Соответствующий Pack | При Close | Экстрактор |
| Крупный урок | `memory/<topic>.md` | Сразу | Claude |

**Формат анонса:** *«Capture: [что] → [куда]»*

### 5.4. CLAUDE.md: как устроен и как настраивать

Система использует два уровня CLAUDE.md:

| Уровень | Файл | Область действия | Кто обновляет |
|---------|------|-----------------|---------------|
| **Корневой** | `~/Github/CLAUDE.md` | Все репо в workspace | Platform (update.sh) + ты (уроки) |
| **Репозиторный** | `<repo>/CLAUDE.md` | Только этот репо | Ты (правила конкретного репо) |

**Когда что использовать:**
- Правило применимо ко всем проектам → корневой CLAUDE.md
- Правило специфично для одного репо → `<repo>/CLAUDE.md`
- Пример: «Всегда pull before commit в DS-strategy» → корневой. «Формат коммитов в DS-aist-bot: feat/fix/chore» → репозиторный.

**Когда ты создаёшь новый DS-* репо**, добавь в него CLAUDE.md с:
- Тип репо (downstream/instrument)
- Связанные Pack (откуда знания)
- Специфичные правила (формат коммитов, тесты, деплой)

## 6. Знания: Pack и экстракция

### 6.1. Что такое Pack

Pack — формализованный паспорт предметной области. **Единственный source-of-truth** для доменного знания.

**Содержит:**
- Bounded Context (границы области)
- Различения (что нельзя путать)
- Онтология (сущности и связи)
- Роли (кто действует)
- Методы (как действовать)
- Рабочие продукты (результаты методов)
- Failure Modes (типичные ошибки)
- SOTA-аннотации (актуальность знания)

**Живой пример:** [PACK-digital-platform](https://github.com/TserenTserenov/PACK-digital-platform) (40+ сущностей)

### 6.2. Создание Pack (11 стадий SPF)

SPF определяет процесс создания Pack:

| # | Стадия | Суть |
|---|--------|------|
| 01 | Выбор домена | Определить и ограничить область |
| 02 | Bounded Context | Установить семантические границы |
| 03 | Работа с различениями | Какие пары нельзя путать |
| 04 | Идентификация сущностей | Роли, объекты, ограничения |
| 05 | Приём информации | Ввод материалов для анализа |
| 06 | Анализ и формализация | Формализация через различения |
| 07 | Извлечение методов и РП | Методы → рабочие продукты |
| 08 | Извлечение failure modes | Типичные ошибки интерпретации |
| 09 | SOTA-аннотации | current / hypothesis / deprecated |
| 10 | Обслуживание карты | Граф связей между сущностями |
| 11 | Цикл ревью и эволюции | Протокол непрерывного обновления |

**Где изучить:**
- [SPF/process/](https://github.com/TserenTserenov/SPF/tree/main/process) — все 11 стадий
- [SPF/pack-template/](https://github.com/TserenTserenov/SPF/tree/main/pack-template) — шаблон структуры

### 6.4. Экстрактор знаний (R2)

**Роль:** Трансформирует информацию в формализованные сущности Pack.

**Пайплайн:** `классифицировать → маршрутизировать → формализовать → валидировать`

| Сценарий | Триггер | Когда ты видишь результат |
|----------|---------|--------------------------|
| Session-Close | Протокол Close | При закрытии сессии Claude предложит новые Pack-сущности |
| On-Demand | Твоя команда | Немедленно в Claude Code |
| Bulk-Extraction | Обработка документов | После анализа — Extraction Report |
| Inbox-Check | Расписание | В DayPlan (если есть новое) |

**Ключевое правило:** Экстрактор всегда **предлагает** — никогда не пишет без одобрения.

**Как это работает для тебя:**
1. Работаешь в Claude Code → по ходу сессии появляются captures
2. При Close → Claude активирует роль Экстрактора (R2) и показывает Extraction Report
3. Ты одобряешь → сущности записываются в Pack

**Где изучить:**
- Протокол Close (§ 5.1) — когда активируется Экстрактор
- [DP.AGENT.001](https://github.com/TserenTserenov/PACK-digital-platform/blob/main/pack/digital-platform/02-domain-entities/DP.AGENT.001-ai-agents.md) R2 — полное описание роли

### 6.5. Knowledge MCP: поиск по базе знаний

Knowledge MCP — это сервис для поиска по всем Pack-репозиториям и документации. 9 источников, ~5400 документов.

**Как использовать (в Claude Code):**

| Команда MCP | Что делает |
|-------------|-----------|
| `knowledge-mcp search` | Семантический поиск по всем источникам |
| `knowledge-mcp get_document` | Получить конкретный документ по имени файла |
| `knowledge-mcp list_sources` | Список всех доступных источников |

**Типы источников:**
- **pack** — доменные знания (PACK-digital-platform, PACK-education и др.)
- **guides** — образовательные гайды
- **ds** — процессы и сценарии

**Пример:** Ищешь, как устроены тиры? → `search("тиры обслуживания", source_type="pack")` → находит DP.ARCH.002.

### 6.6. Онтология: граф знаний

Онтология — граф понятий и связей. Каждый уровень имеет свою:

| Уровень | Где | Что |
|---------|-----|-----|
| SPF-level | `SPF/ontology.md` | Универсальные понятия фреймворка |
| Pack-level | `PACK-{}/01-domain-contract/01C-ontology.md` | Сущности конкретного Pack |
| Экосистемная | `DS-ecosystem-development/ontology.md` | 31 понятие: платформа + экосистема |
| Личная | `DS-strategy/ontology.md` | Личное развитие + кросс-связи |

**Принцип:** SPF наследует FPF → Pack расширяет SPF → Downstream ссылается на Pack.

## 7. Роли и ИИ-агенты

### 7.1. Ролецентричный подход (DP.D.033)

В IWE роль описывается **независимо от исполнителя**. Сначала: что делать, какие обязательства, какие рабочие продукты. Потом: кто исполняет.

| Понятие | Определение |
|---------|-----------|
| **Роль** | Функция: ЧТО делать (обязательства, РП, методы) |
| **Исполнитель (holder)** | Система: КТО делает (Claude, bash, человек) |
| **Агент** | Исполнитель с автономностью (Grade 2+) |
| **Инструмент** | Исполнитель без автономности (Grade 0-1) |

**Ключевые принципы:**
- **Роль ≠ Система.** Одно имя может обозначать и роль, и систему — это разные ракурсы
- **Один исполнитель — много ролей.** Claude играет роли Стратега, Экстрактора, Кодера
- **Одна роль — много исполнителей.** Роль Синхронизатора: bash (механика) + Claude (аудит)

**Нотация:** `Holder#Role:Context@Window` (FPF A.2)

### 7.2. Каталог агентов

#### У тебя (L4 Personal IWE)

Эти агенты работают в твоём Claude Code, на твоей машине:

| Агент | Роль | Что делает | Когда видишь результат |
|-------|------|-----------|----------------------|
| **Стратег (R1)** | Планирование | WeekPlan, DayPlan, стратегические сессии | Утро (launchd → Telegram), сессия (Claude Code) |
| **Экстрактор (R2)** | Формализация знаний | Captures → Pack-сущности | Close сессии в Claude Code |

#### На платформе (L2 Platform)

Эти агенты работают на инфраструктуре платформы. Ты видишь только результаты:

| Агент | Роль | Что делает | Как ты видишь результат |
|-------|------|-----------|------------------------|
| **Синхронизатор (R8)** | Координация | Fleeting-notes sync, уведомления | Telegram-уведомления |
| **Шаблонизатор (R9)** | Обновление шаблона | Drift detection, validation | При `update.sh` |
| **Статистик (R10)** | Аналитика | DAU/WAU/MAU, retention | `/analytics` в боте |
| **Наладчик (R11)** | Исправление ошибок | Auto-fix, restart, escalate | GitHub Issues, TG-уведомления |

> **Для T4 пользователя:** R1 (Стратег) и R2 (Экстрактор) — основные. Платформенные агенты (R8-R11) работают в фоне.

### 7.3. Схема взаимодействия агентов

```
Расписание / Действие пользователя
    ↓
R8 Синхронизатор (диспетчер)
    ├─→ R1 Стратег (планы, обзоры)
    │   └─→ DS-strategy/current/Plan, Day, Report
    ├─→ R2 Экстрактор (знания)
    │   └─→ Pack-сущности, DS-strategy/inbox/
    ├─→ R9 Шаблонизатор (обновления)
    │   └─→ FMT-exocortex-template/
    ├─→ R11 Наладчик (ошибки бота)
    │   └─→ GitHub PR, Issues
    ├─→ R10 Статистик (аналитика)
    │   └─→ TG-отчёт, /analytics
    └─→ Уведомления в Telegram
```

**Где изучить:**
- Каталог ролей (21 роль R1-R21): [DP.AGENT.001](https://github.com/TserenTserenov/PACK-digital-platform/blob/main/pack/digital-platform/02-domain-entities/DP.AGENT.001-ai-agents.md)

## 8. Качество и архитектура решений

### 8.1. ArchGate (ЭМОГСС)

**Блокирующее правило:** Любое архитектурное решение оценивается по 6 характеристикам. Порог ≥8.

| Характеристика | Вопрос |
|----------------|--------|
| **Э**волюционируемость | Что сломается при изменении? |
| **М**асштабируемость | Что будет при 10x? |
| **О**бучаемость | Сколько читать, чтобы начать? |
| **Г**енеративность | Создаёт ли платформу для нового? |
| **С**корость | Какая задержка? (бот <3 сек, CLI <1 сек) |
| **С**овременность | Как решают лучшие? Проверь SOTA. |

**Формат:** Решение → таблица оценки → что слабо → как усилить.

### 8.2. SOTA-практики

Приоритетная тройка (проверяй ВСЕГДА при архитектурных решениях):

| # | Практика | Суть |
|---|----------|------|
| 1 | Context Engineering | Write/Select/Compress/Isolate — что попадает в контекст агента |
| 2 | DDD Strategic | BC = Pack scope, UL = ontology, Context Map = typed `related:` |
| 3 | Coupling Model | Связи по 3 измерениям: knowledge, distance, volatility |

Полный список: 12 платформенных + 6 Pack-архитектурных = 18 практик.

**Где изучить:**

## 10. Рост и развитие

### 10.1. Создание своего Pack

**Когда создавать:**
- Регулярно работаешь в одной области
- Важно не терять знания между сессиями
- Хочешь, чтобы Claude знал термины и паттерны твоей области

```bash
# 1. Клонируй SPF (read-only reference)
gh repo clone TserenTserenov/SPF ~/Github/SPF

# 2. Создай Pack из шаблона
cp -r ~/Github/SPF/pack-template ~/Github/PACK-my-domain
cd ~/Github/PACK-my-domain
git init && git add -A && git commit -m "Initial Pack: my-domain"
gh repo create PACK-my-domain --private --source=. --push
```

Затем откройте Claude Code — он проведёт через 11 стадий SPF.

### 10.2. Новые агенты и инструменты

Перед добавлением — IntegrationGate (§ 8.4). После определения:

| Компонент | Тип | Где описание | Где реализация |
|-----------|-----|-------------|---------------|
| Экстрактор | Агент (Grade 2) | DP.AGENT.001 R2 | DS-ai-systems/extractor/ |
| Синхронизатор | Агент + Инструмент | DP.AGENT.001 R8 | DS-ai-systems/synchronizer/ |

**Принцип:** Минимальная сложность на старте. Стратег один достаточен для первых месяцев. Экстрактор — когда Pack достигнет 10+ сущностей. Синхронизатор — когда 3+ репозитория.

### 10.3. MAPSTRATEGIC.md: стратегия каждой системы

Когда создаёшь новый репо, добавь `MAPSTRATEGIC.md`:

```markdown
# MAPSTRATEGIC: {Название системы}

## Текущая фаза
{Описание: какие задачи решаются сейчас}

## Следующая фаза
{Куда двигается система}

## Горизонт
{Долгосрочное видение}
```

Стратег читает все MAPSTRATEGIC при Session-Prep и агрегирует в `Strategy.md`.

### 10.4. Как развивать IWE самостоятельно

**Принцип:** Начни с минимума, усложняй по мере роста.

```
День 1:       setup.sh → FMT (fork) + DS-strategy          ← старт
Неделя 1:     Ежедневная работа с Claude Code + Стратег     ← привычка
Неделя 2-4:   Первый PACK-{область}                        ← формализация знаний
Месяц 2-3:    DS-{проекты} (код, контент)                  ← созидание
По мере роста: Экстрактор, Синхронизатор, свои Format      ← масштабирование
```

**Рекомендации:**
- **Не клонируй** все репозитории сразу — начни с FMT + DS-strategy
- **Не создавай Pack** пока не определил область и не накопил captures
- **Не добавляй агентов** пока не справляешься без них (IntegrationGate, § 8.4)
- **Клонируй SPF** только когда готов создать Pack (read-only reference)
